# Category 5 тАФ AI

**Status:** тЬЕ Fully documented (extra-detailed) ┬╖ **Last verified against code:** 2026-06-12
**Owner:** Michael Bernard

The intelligence layer: which models, what we ask them, how accuracy is enforced, what it costs per token, and where the product is heading on providers. Category 2 owns the *pipeline mechanics* (PDF in, marked PDF out); this category owns **what the model sees, what we tell it, and what it returns**.

The one-line truth: **a single Anthropic call per paper, marking extracted text (not images) against a memo, returning short-keyed JSON, with the system + memo prompt-cached to cut cost.** Everything else (Gemini, Ollama, OCR) is roadmap, not built.

---

## 1. Models & the hidden "quality" abstraction

Users never see model names. The UI offers **Standard** vs **High accuracy** (Category 3 ┬з5.1); the mapping is server-side only:

```ts
// lib/markingPrompt.ts
export const MODELS = {
  standard: "claude-sonnet-4-6",   // default
  high:     "claude-opus-4-7",     // "High accuracy" toggle
} as const;
```

- The request carries `quality: "standard" | "high"`; the route resolves `MODELS[quality] ?? MODELS.standard` (Sonnet is the safe fallback for any unknown value).
- **Opus burns allowance ~5├Ч faster** than Sonnet (see the cost table ┬з6) тАФ which is why the SettingsPanel copy says "Higher accuracy uses more of your allowance" without naming the model.
- **Why hide names:** the percentage-only allowance model (Cat 1) and the ability to swap models silently both depend on the UI never committing to a model string. Changing the map here changes the product with zero UI churn.

This matches ADR-001's recommendation exactly: **Sonnet default, Opus reserved for a high-accuracy toggle.**

## 2. Input strategy тАФ text-first, image fallback

This is the single most important AI decision, and it **diverged from the original ADR**:

- **ADR-001 (2026-05-26)** was written assuming **handwritten scans** тЖТ vision-first (send page images). Its provider/model section is still "Proposed."
- **The update at the top of ADR-001** records the pivot: inputs are confirmed **typed**, so the **accepted** strategy is **extract text + positions from the PDF text layer тАФ no vision, no OCR** for the common case.

The implementation (Cat 2 ┬з3) reflects the pivot: `preparePaper` emits cheap **text** pages with `[y=0.NN]` position hints, and only falls back to a **base64 PNG image block** for pages with no text layer (scans/diagrams). So in practice:

- **Typed paper (the target market)** тЖТ all text blocks тЖТ cheapest path, best placement.
- **Scanned page** тЖТ image block тЖТ expensive vision tokens, less precise placement.

The cost rationale (`cost-and-pricing-notes.md`): one page as an **image тЙИ 1,000тАУ2,000+ input tokens**; the same page as **text тЙИ a few hundred** тАФ roughly a **10├Ч difference**. Rendering and stamping are free (client-side); the *only* variable cost is how much the model has to read.

## 3. The marking prompt тАФ `buildSystem(strictness, markTypes, subject)`

Returns a single **cached** `TextBlockParam`. Anatomy, block by block:

1. **Role line** тАФ "You are an exam marker for university `{subject}` tests. Your single job is to mark each student answer against the supplied MEMOтАж and award marks exactly as the memo dictates." (`subject` is interpolated when set.)
2. **HOW TO MARK** тАФ
   - *The memo is the only source of truth.* Compare to the memo and nothing else.
   - *Reward accuracy, never style* тАФ a short correct answer earns full marks; a long eloquent wrong one earns nothing.
   - *Partial marks per memo-allocated point present in the answer.*
   - **Strictness** injected verbatim: `Apply strictness ${strictness}/10 (1 = lenient тАж 10 = strict).`
3. **DO NOT HALLUCINATE** (the trust guard) тАФ never invent facts/marks/questions; no marks for plausible-but-unsupported content; missing/unreadable/off-topic тЖТ mark 0 and say so; every mark must be justifiable from answer + memo.
4. **INPUT FORMAT** тАФ explains the `[y=0.NN]` line prefixes (0 = top, 1 = bottom) and that some pages may be images to read directly.
5. **OUTPUT contract** тАФ the short-key schema (┬з5) built from the user's mark types: `shapeList = markTypes.map(m => "\"${shape}\" (${abbrev} = ${label})")`. So the **available shapes are user-defined** (Cat 3 SettingsPanel), passed into the prompt.
6. **Strict JSON shape** тАФ an exact example object, "no prose outside it."

> **Design intent:** accuracy-first, anti-hallucination, memo-anchored. The prompt is deliberately strict because, per ADR-001's trade-off analysis, *"a cheap wrong mark is worse than an expensive right one."*

## 4. Content assembly & prompt caching тАФ `buildContent(memoText, pages)`

Builds the user-turn content array:

1. **MEMO block** тАФ `MEMO (answer key):\n${memoText || "(none provided)"}`, marked **`cache_control: { type: "ephemeral" }`**.
2. **One block per page** тАФ text pages as `--- Page N ---\n[y=тАж] тАж`; image pages as a base64 PNG `image` block.
3. **Trailing instruction** тАФ "Mark the student's answers above against the memo. Return only the JSON."

**Prompt caching is the key cost lever here.** Two blocks are cached ephemeral:
- the **system prompt** (`buildSystem`), and
- the **memo block** (`buildContent`).

Across a batch of papers marked against the *same memo with the same settings*, those two large, identical blocks are **cache reads** (Sonnet $0.30/M vs $3/M input тАФ 10├Ч cheaper; Opus $1.50/M vs $15/M). Only the per-paper page text is fresh input. This is why `lib/cost.ts` tracks `cache_creation_input_tokens` and `cache_read_input_tokens` separately (Cat 1 ┬з5) тАФ the savings are real and metered.

> The instant and batch routes share `buildSystem`/`buildContent` verbatim, so caching behaves identically across both modes and the two can't drift.

## 5. The short-key output contract & parsing

To save output tokens, Claude is told to emit **short keys**, not full field names:

| Short | Meaning | Expands to |
|-------|---------|-----------|
| `p` | page number (from 1) | `page` |
| `y` | vertical position 0.0тАУ1.0 | `y` |
| `s` | shape symbol (one of the mark types) | `shape` |
| `m` | awarded/available, e.g. `"3/5"` | `marks` |
| `c` | margin note (тЙд ~8 words) | `comment` |

Plus top-level `total`, `available`, `percentage`, `summary`.

**`parseMarkResponse(raw)`** is the robustness layer (also Cat 2 ┬з4):
1. Prefer a fenced ` ```json тАж ``` ` block if present.
2. Else slice from the first `{` to the last `}` тАФ tolerates any prose the model wraps around the JSON ("I don't see student answersтАж {тАж}").
3. `JSON.parse`, then **expand short keys тЖТ full `Annotation`** with defaults (`p ?? 1`, `y ?? 0`, `s ?? "tick"`, `m ?? ""`, `c ?? ""`).
4. Throws `"No JSON object foundтАж"` if there are no braces (handled per-paper in batch, bubbles to 500 in instant тАФ Cat 4 ┬з3.2/┬з3.3).

## 6. Cost model (AI view)

Full detail is in **Category 1 ┬з5**; the AI-relevant facts:

| Model | input | output | cache write | cache read |
|-------|-------|--------|-------------|------------|
| `claude-sonnet-4-6` | $3/M | $15/M | $3.75/M | $0.30/M |
| `claude-opus-4-7` | $15/M | $75/M | $18.75/M | $1.50/M |

- `USD_TO_ZAR = 18.5`. Unknown model тЖТ Sonnet rates.
- **Per-paper estimates** (used for batch pre-flight, Cat 1 ┬з6.2): Sonnet тЙИ **R0.69**, Opus тЙИ **R3.50**.
- **Levers** (from `cost-and-pricing-notes.md`): (1) text extraction for typed PDFs тАФ *built*; (2) OCR-then-reason тАФ *not built* (┬з8); (3) lower-res image renders + skip blank pages тАФ *partially built* (blank-skip done at scale 1.6, Cat 2 ┬з3.1).
- **`max_tokens: MAX_OUTPUT_TOKENS` (16000)** is set on every marking call (both routes, shared constant in `lib/markingPrompt.ts`) тАФ a ceiling on output cost per paper. Raised from 4096 for headroom against truncation; only generated tokens are billed, so the higher cap costs nothing on normal papers. 16k тЙИ 400+ annotations (2тАУ4├Ч a heavy paper) and only output counts, so a realistic test won't hit it; an overflow is detected (`stop_reason`) and reported, never silently half-marked. **P2-4 / P5-3 closed** on this basis (chunking declined). тЪая╕П If thinking is ever enabled here, thinking tokens count against the 16k тАФ bump the cap.

## 7. Provider strategy & roadmap (ADR-001)

The decision: **a thin `MarkingProvider` interface; launch on Claude; slot in Gemini and Ollama behind it.** The interface and adapters are **not built yet** тАФ today there is a single hard-coded Anthropic call in each route.

Intended mapping of providers тЖТ pricing tiers:

| Provider | Cost to us | Privacy (POPIA) | Status | Tier |
|----------|-----------|-----------------|--------|------|
| **Claude (we provide)** | We pay per token | Paid tier, no training | тЬЕ Built | "Done-for-you" (Standard/Pro) |
| **Gemini (BYO key)** | User pays their provider | Depends on their tier | тмЬ Planned | ~R200 BYO tier |
| **Ollama (local)** | Free (their hardware) | Excellent тАФ data never leaves device | тмЬ Planned | Local/private tier |

Architecture note (from the cost notes): a **server can't reach a user's `localhost`**, so an Ollama option must call `localhost:11434` **from the browser** тАФ feasible because rendering/stamping/file IO are already client-side. Needs `OLLAMA_ORIGINS` config.

Proposed interface (ADR-001 action item, not yet implemented):
```ts
interface MarkingProvider {
  mark(pages, memo, markTypes, strictness): Promise<annotations + score>
  estimateCost(...): number
}
// AnthropicProvider (refactor current call) тЖТ GeminiProvider тЖТ OllamaProvider
```

## 8. OCR-then-reason (the big future cost lever)

Status: **not built** (Phase 5 / ADR-001 Option C, deferred).

Idea: run cheap/local **Tesseract** OCR on an image page тЖТ get **text + word bounding boxes** тЖТ feed only clean **text** to the expensive model (cheap tokens) тЖТ **stamp at the OCR boxes** (precise placement, better than a vision model guessing coordinates).

Two independent wins, even on the Claude tier:
1. **Cost** тАФ image тЖТ text collapses the 10├Ч token gap for scanned pages.
2. **Placement accuracy** тАФ OCR boxes beat vision-guessed `y` values. ADR-001 calls this "the sleeper issue": worth adopting for *where* to stamp even if reasoning stays on Claude.

Catch: handwriting OCR is unreliable тАФ which is exactly why it was deferred behind the "inputs are typed" pivot.

## 9. Diagnostic tooling (AI debugging)

Two Node scripts reproduce the AI path **outside the app** against the local test pack (`C:\Users\Michael Bernard\TestPapers`):

- **`scripts/extract-debug.mjs`** тАФ replicates `preparePaper`'s per-page text extraction (same `[y=тАж]` line-grouping, 0.012 threshold) for `Student_A/B.pdf` and prints it. Use to verify *what text the model will actually see*.
- **`scripts/debug-mark.mjs`** тАФ does a full marking call against the test PDFs (Sonnet, cached system + memo) and prints Claude's raw reply (first 400 chars). Use to debug *prompt тЖТ response* without the browser.

> тЪая╕П **`debug-mark.mjs` carries a frozen, slightly stale copy of the prompt** (hard-coded "Taxation" subject, only 3 shapes, strictness 7, `max_tokens: 1500`). It does **not** import `buildSystem`, so it can drift from the live prompt. Treat it as a throwaway repro harness, not a source of truth тАФ the live prompt is `lib/markingPrompt.ts`.

## 10. Known gaps & not-built

- **No `MarkingProvider` abstraction** тАФ single hard-coded Anthropic call per route. Refactoring this is the unlock for everything below.
- **No Gemini / Ollama adapters** тАФ the BYO and local/private tiers don't exist.
- **No OCR-then-reason** тАФ scanned pages still cost full vision tokens with approximate placement.
- **No model-quality A/B** тАФ ADR-001's action item to verify Sonnet-vs-Opus marking parity hasn't been run; the "high accuracy" claim is assumed, not measured.
- **No measured cache-hit rate** тАФ caching is wired and metered, but nobody's confirmed the hit rate in practice (Cat 4's missing observability).
- **Prompt is single-shot** тАФ no self-check / second-pass / confidence scoring; one call decides the marks.
- **`mockResult()` exists but is dead** тАФ routes 503 without a key rather than fabricate marks (Cat 2 ┬з4).

## 11. Invariants тАФ do not break these

1. **Model names stay server-side.** The UI says Standard/High; never leak `claude-*` to the client. The whole pricing/abstraction model depends on it.
2. **Memo is the only source of truth** in the prompt, and **anti-hallucination rules stay** тАФ accuracy is the product.
3. **Keep system + memo blocks `cache_control: ephemeral`.** Removing caching multiplies input cost ~10├Ч on batches.
4. **Instant and batch must share `buildSystem`/`buildContent`** тАФ never fork the prompt between modes.
5. **`max_tokens` cap stays** тАФ it bounds per-paper output cost.
6. **Quality fallback is Sonnet**, never Opus тАФ an unknown `quality` must not silently 5├Ч the cost.
