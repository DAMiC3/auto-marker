Everything the user sees and every message we send them. This category owns the **visual system, components, layout, theming, PWA shell, and all user-facing communication**. **Notifications fold in here** тАФ batch-complete emails, trial-expiry warnings, renewal confirmations are user-facing communication, so they live with the UI even though the infrastructure isn't built yet.

## 1. Design language (the visual system)

AutoMark is a **two-zone layout**: a **dark navy sidebar** and a **light app body**.

| Token | Value | Used for |
|-------|-------|----------|
| Sidebar base | `#0E1525` | Left nav background |
| Sidebar logo strip | `#090E1A` | Top-left logo block |
| Sidebar popovers/cards | `#161E2E` | Menus, the about popover, login card |
| App body | `#F3F6FB` | Main content background |
| Cards | `white` + `border-slate-200`, `rounded-2xl` | Every content panel |
| Neutral text | Tailwind `slate-*` | Body copy, labels |
| Accent | `--accent-50тАж700` CSS vars | Buttons, highlights, active states |
| Font | **Inter** (`next/font/google`, `--font-inter`) | Everything |

**Conventions seen throughout:**
- Pixel-locked type sizes (`text-[11px]` тАж `text-[28px]`) rather than Tailwind's named scale тАФ tight, deliberate typography.
- Section labels: `text-[12px] font-medium text-slate-500`.
- Primary buttons: `bg-[var(--accent-600)] hover:bg-[var(--accent-700)] text-white`.
- Rounded everything (`rounded-lg/xl/2xl`), generous padding, hairline `border-slate-200`.
- Inline SVG icons (stroke-width 2тАУ2.5), no icon library.

## 2. Theming system (accent colours)

A runtime-switchable accent drives the whole palette via CSS variables.

- **5 accents:** `indigo` (default), `violet`, `emerald`, `rose`, `amber`. Defined in [`app/globals.css`](../../app/globals.css) as `[data-accent="тАж"]` blocks, each exposing six shades (`--accent-50/100/400/500/600/700`) mirroring Tailwind's palette.
- **Where it's stored:** `localStorage["automark.settings"].accent` (a key string, not a hex).
- **How it's applied:** the `data-accent` attribute on `<html>`.
- **Flash-free application:** [`app/layout.tsx`](../../app/layout.tsx) runs a `beforeInteractive` inline `<Script id="accent-init">` that reads the saved accent from localStorage and sets `document.documentElement.dataset.accent` **before paint**. On change, `SettingsPanel.handleSave` sets the same attribute immediately.
- **Adding an accent requires two edits in lockstep:** a new `[data-accent]` block in `globals.css` **and** a new entry in the `ACCENTS` array in `SettingsPanel.tsx`. Miss either and the swatch shows but the colours don't change (or vice-versa).

`globals.css` is minimal: `@import "tailwindcss";` (Tailwind v4), `@theme inline { --font-sans: var(--font-inter); }`, a `height:100%` reset, and the five accent blocks. No other global styles.

## 5. Components catalogue

### 5.1 `SettingsPanel.tsx` тАФ the settings slide-over (also the settings *model*)

This file is **double-duty**: it renders the panel **and** defines the canonical `Settings` type, defaults, and persistence used app-wide.

**Exported model:**
- `interface Settings { defaultStrictness, accent, profile{name,subject}, markTypes[], markingQuality, subjects[] }`
- `DEFAULT_SETTINGS` тАФ strictness 7, accent `indigo`, profile `{name:"Michael Bernard", subject:"English"}`, `DEFAULT_MARK_TYPES`, quality `standard`, subjects `["English","Mathematics"]`.
- `DEFAULT_MARK_TYPES` тАФ six exam-standard marks: **M** Full (green tick), **┬╜** Half (green half), **A** Accuracy (blue tick), **тЬЧ** Incorrect (red cross), **E** Explanation (amber circle), **FT** Follow-through (cyan underline).
- `loadSettings()` / `saveSettings()` тАФ read/write `localStorage["automark.settings"]`, with defensive merging against `DEFAULT_SETTINGS` (so older saved blobs don't break when the shape grows).

**Panel UI** (right slide-over, `max-w-md`, `slidein` keyframe, backdrop blur):
- **Profile** тАФ display name + subject inputs.
- **Default marking strictness** тАФ 1тАУ10 range with a `n/10` pill.
- **Marking engine** тАФ two cards, **Standard** vs **High accuracy**. *Model names are intentionally hidden* (the Sonnet/Opus mapping lives server-side, Cat 5); copy only says "Higher accuracy uses more of your allowance."
- **Mark types** тАФ full CRUD: live `MarkShapeIcon` preview, native colour picker, abbrev (`maxLength 3`), label, remove, a 6-shape picker, and "Add mark type". This is what the PDF stamper (Cat 2) draws and what the marking prompt (Cat 5) lists.
- **Accent color** тАФ five swatches (the `ACCENTS` array).
- **Account** тАФ sign out.
- **Footer** тАФ Cancel / Save. Save persists to localStorage, applies `data-accent`, calls `onSave`, closes.

> Note: `handleSave` preserves `initial.subjects` untouched тАФ **subjects are managed elsewhere** (the combobox on the main page, ┬з5.5), not in this panel.

### 5.2 `MarkShapeIcon.tsx` тАФ on-screen shape renderer
Exports `MarkShape` type and `MARK_SHAPES` list (tick/half/cross/circle/underline/dot) and renders each as an inline SVG. Used for previews and the shape picker in SettingsPanel.

> тЪая╕П **Two separate shape implementations exist.** This file draws shapes *on screen*; `drawShape()` in `lib/markPaper.ts` (Cat 2) draws them *onto the PDF* with different primitives. They must be kept **visually consistent by hand** тАФ changing one doesn't change the other.

### 5.3 `AllowanceBar.tsx`
Category 1 surface (see that doc ┬з8.1). Visually: plan label + "% left", progress bar, тЙд15% red low-balance warning, "Buy a plan to start marking тЖТ" when cap is 0. Listens for the `allowance-refresh` window event.

### 5.4 `StrictnessSlider.tsx`
Standalone card with a 1тАУ10 accent range and LenientтЖФStrict end labels. Mirrors the SettingsPanel strictness control but is the **per-batch** value on the main page (seeded from `defaultStrictness`, overridable per run).

### 5.5 `SubjectCombobox.tsx`
Custom searchable combobox (no library). Filters `options` by query, supports **add-new** ("Add '<x>'"), click-away to close, and Enter to either add the typed value or pick the first match. New subjects flow up via `onAddOption` тЖТ `handleAddSubject` in `page.tsx` тЖТ persisted into `settings.subjects`.

## 6. PWA shell (install + update)

AutoMark is a **Progressive Web App** тАФ installable, with a service worker and an update flow.

- **Manifest & metadata** тАФ [`app/layout.tsx`](../../app/layout.tsx): `manifest: "/manifest.json"`, `appleWebApp` (capable, black-translucent, title "AutoMark"), `viewport.themeColor: "#4F46E5"`.
- **`InstallButton.tsx`** (header) тАФ captures Chrome's `beforeinstallprompt`, suppresses the mini-infobar, and shows an "Install app" button **only when installable**. Hides if already installed (`appinstalled` event or `display-mode: standalone`).
- **`UpdatePrompt.tsx`** (rendered globally in `layout.tsx`) тАФ registers `/sw.js`, detects a waiting service worker, shows a centered **"A new version is available. тЖТ Reload"** pill, posts `SKIP_WAITING`, and reloads once on `controllerchange`. This is the app's only persistent toast-style notification.

### 8.3 Ops alerts тАФ `notifyOps()` (BUILT 2026-06-15)

The first **real outbound notification channel** in the app. It alerts the founder (not the end-user) when something operational fails тАФ currently (a) a Supabase usage-**write** failure (Category 1, Problem 8) and (b) a usage-**read/verify** failure that blocked a user from marking (Category 1, P1-2 тАФ `checkAllowance()`).

- **Code:** `lib/notify.ts` тАФ `notifyOps(message)`. Always `console.error`s (so it's in Cloudflare observability), and if the `OPS_ALERT_WEBHOOK_URL` secret is set, `POST`s the message there.
- **Channel:** a plain-text webhook. Designed for a free **ntfy.sh** topic (subscribe in the ntfy app тЖТ phone push). Slack/Discord work too, but expect a JSON body (`{"text":тАж}` / `{"content":тАж}`) тАФ tweak the `fetch` in `notify.ts` if using those.
- **Setup:** `npx wrangler secret put OPS_ALERT_WEBHOOK_URL` тЖТ paste the topic URL. **Until set, it's log-only** тАФ no breakage, just no push.
- **Fired from:**
  1. `recordUsage()`'s final-failure path (`lib/usage.ts`), alongside parking the event in the D1 dead-letter buffer (Cat 1 ┬з11b/┬з13b). *Message:* `Supabase usage write failed тАФ parked RтАж тАж`.
  2. `checkAllowance()`'s verification-failure paths (`lib/usage.ts`, Cat 1 ┬з6.3) тАФ when an auth lookup or profile read errors and a user is therefore **blocked from marking**. *Message:* `Marking blocked: could not read allowance for <user> тАФ тАж`. (The normal over-limit case does **not** page тАФ that's expected, not a fault.)
  3. **`/api/report-error`** (`app/api/report-error/route.ts`, P3-8) тАФ fired by the client whenever the UI hits an **unrecognized** error (the `friendlyError` generic-fallback path). The route attaches *who* hit it from the session cookie and pushes *`error hit by <email> during "<context>": <detail>`*, so unexpected client-side failures surface (with user + where) instead of dying silently in the browser. The user only ever sees the generic message.
- **Not yet wired:** end-user notifications (batch-complete, trial-expiry) in ┬з8.2 are still unbuilt. `notifyOps` is reusable for any future ops alert.

### 8.4 User-facing block & plan notifications (BUILT)

When marking is refused, the user is **always told why** тАФ never left with a silent no-op or a raw error code. These are the user-facing counterpart to the ┬з8.3 ops alerts. The *decision* logic lives in Category 1 (`lib/allowance.ts` + `checkAllowance()`); the *presentation* lives here.

| Trigger | What the user sees | Surface / code |
|---------|--------------------|----------------|
| **Marking blocked by a system error** (auth/DB couldn't be verified тАФ fail-closed, P1-2) | Red banner: *"We couldn't verify your plan just now (a temporary system error), so nothing was marked. Please try again in a moment тАФ you weren't charged."* | `friendlyError("verification_failed")` тЖТ error banner in `app/page.tsx` |
| **Session expired** | Red banner: *"Your session has expired. Please sign in again to keep marking."* | `friendlyError("not_authenticated")` |
| **Allowance used up** | Red banner: *"You've used up your plan's allowance. Buy another plan to keep marking."* | `friendlyError("allowance_exhausted")` |
| **Unrecognized error** (anything we have no specific message for, P3-8) | Red banner: *"An error occurred. Please try again later. If this keeps happening, please contact the developer at &lt;dev email&gt;."* + a **founder alert** (┬з8.3) | `friendlyError()` generic fallback + `reportError()` in `app/page.tsx` |
| **Plan expired / limit reached** (persistent, not tied to a mark attempt) | Top-of-page banner *"Your plan has expired / You've reached your plan limit"* + **Buy a new plan** button | `components/PlanNotice.tsx` (Cat 1 ┬з8.3) |
| **Plan expiry date** (informational) | Settings тЖТ Plan: *"Renews/expires on &lt;date&gt;"* | `components/SettingsPanel.tsx` (Cat 1 ┬з8.4) |

`friendlyError(raw)` (formerly `blockMessage`) is the single mapping from a thrown error to a human sentence. It returns `{ message, recognized }`: known codes/patterns get a specific sentence (`recognized: true`); anything else gets a **generic fallback** тАФ *"An error occurredтАж contact the developer at &lt;dev email&gt;"* тАФ with `recognized: false`. **Every** catch path in `app/page.tsx` runs through it (no more raw `setError(e.message)`), and on `recognized: false` it also calls `reportError()` to page the founder (┬з8.3). Add new known errors there so users get guidance and the founder isn't paged for expected cases.

## 9. Cross-cutting UX behaviours

- **Hydration safety** тАФ `page.tsx` assumes `supported=true` during SSR, then re-checks `isSupported()` after mount via a `mounted` flag, to avoid a server/client mismatch on the File System Access API check.
- **Flash-free theming** тАФ accent applied before paint (┬з2).
- **Persistence split** тАФ settings & accent in `localStorage`; folder handles & memos in IndexedDB (Cat 2). Nothing user-specific is server-stored except the profile/usage rows (Cat 1).
- **Event bus (lightweight)** тАФ the only cross-component signal is the `allowance-refresh` `window` event fired after marking, which tells `AllowanceBar` to re-fetch.
- **Strictness lives in two places** тАФ the per-batch `StrictnessSlider` (seeded from `defaultStrictness`) and the SettingsPanel default. Changing the panel default reseeds the page slider on save.
- **Banner feedback (P3-5)** тАФ the red/green banners are **dismissible** (close button) and **auto-dismiss** (success 8 s, error 15 s); a run's **Results card is persisted** to `localStorage["automark.lastResults"]` and restored on mount, so a refresh doesn't blank it. Async actions (connect, add-memo) show inline loading text + disable while busy.
- **Slide-over accessibility (P3-6)** тАФ `SettingsPanel` is a `role="dialog" aria-modal` with a **focus trap**, **Esc to close**, focus-in on open and focus-restore to the trigger on close. The sidebar popovers (about, account menu) also close on **Esc** and expose `aria-expanded`/`role="menu"`.

## 10. Known gaps & issues (UI)

- **No onboarding / help system.** First-run is just the empty "Connect your files" state тАФ no guided tour, no help button. This is **Phase 1** of the expansion plan (big help button on first launch тЖТ shrinks to a corner). Currently the single biggest UX gap.
- **Two shape implementations** (`MarkShapeIcon` on screen vs `drawShape` on PDF) drift independently.
- **Chrome/Edge-only** is surfaced as an amber notice, but non-Chromium users have no path forward until the Phase-1 picker change (Cat 2).
- **No notification infrastructure** (┬з8.2) тАФ lecturers must watch the tab during batch runs.
- **Wording is partially lecturer-friendly** ("Memo (answer key)", "Marking engine" instead of model names) but a full copy audit is still a Phase-1 item.
- **Accent/subjects coupling is implicit** тАФ adding an accent needs paired edits in CSS + `ACCENTS`; subjects are edited in the combobox, not Settings. Easy to trip over.
