# Category 3 — UI (incl. Notifications & Communication)

**Status:** ✅ Fully documented (extra-detailed) · **Last verified against code:** 2026-06-12
**Owner:** Michael Bernard

Everything the user sees and every message we send them. This category owns the **visual system, components, layout, theming, PWA shell, and all user-facing communication**. **Notifications fold in here** — batch-complete emails, trial-expiry warnings, renewal confirmations are user-facing communication, so they live with the UI even though the infrastructure isn't built yet.

Boundaries: the *logic* behind a surface lives in its own category (allowance maths → Cat 1, the marking flow's mechanics → Cat 2, auth → Cat 7). This doc covers how those surfaces **look, are structured, and behave on screen**.

---

## 1. Design language (the visual system)

AutoMark is a **two-zone layout**: a **dark navy sidebar** and a **light app body**.

| Token | Value | Used for |
|-------|-------|----------|
| Sidebar base | `#0E1525` | Left nav background |
| Sidebar logo strip | `#090E1A` | Top-left logo block |
| Sidebar popovers/cards | `#161E2E` | Menus, the about popover, login card |
| App body | `#F3F6FB` | Main content background |
| Cards | `white` + `border-slate-200`, `rounded-2xl` | Every content panel |
| Neutral text | Tailwind `slate-*` | Body copy, labels |
| Accent | `--accent-50…700` CSS vars | Buttons, highlights, active states |
| Font | **Inter** (`next/font/google`, `--font-inter`) | Everything |

**Conventions seen throughout:**
- Pixel-locked type sizes (`text-[11px]` … `text-[28px]`) rather than Tailwind's named scale — tight, deliberate typography.
- Section labels: `text-[12px] font-medium text-slate-500`.
- Primary buttons: `bg-[var(--accent-600)] hover:bg-[var(--accent-700)] text-white`.
- Rounded everything (`rounded-lg/xl/2xl`), generous padding, hairline `border-slate-200`.
- Inline SVG icons (stroke-width 2–2.5), no icon library.

## 2. Theming system (accent colours)

A runtime-switchable accent drives the whole palette via CSS variables.

- **5 accents:** `indigo` (default), `violet`, `emerald`, `rose`, `amber`. Defined in [`app/globals.css`](../../app/globals.css) as `[data-accent="…"]` blocks, each exposing six shades (`--accent-50/100/400/500/600/700`) mirroring Tailwind's palette.
- **Where it's stored:** `localStorage["automark.settings"].accent` (a key string, not a hex).
- **How it's applied:** the `data-accent` attribute on `<html>`.
- **Flash-free application:** [`app/layout.tsx`](../../app/layout.tsx) runs a `beforeInteractive` inline `<Script id="accent-init">` that reads the saved accent from localStorage and sets `document.documentElement.dataset.accent` **before paint**. On change, `SettingsPanel.handleSave` sets the same attribute immediately.
- **Adding an accent requires two edits in lockstep:** a new `[data-accent]` block in `globals.css` **and** a new entry in the `ACCENTS` array in `SettingsPanel.tsx`. Miss either and the swatch shows but the colours don't change (or vice-versa).

`globals.css` is minimal: `@import "tailwindcss";` (Tailwind v4), `@theme inline { --font-sans: var(--font-inter); }`, a `height:100%` reset, and the five accent blocks. No other global styles.

## 4. Sidebar — `components/Sidebar.tsx`

The persistent 260px dark navigation. Props: `folders, activeFolder, connected, profileName, profileSubject, onConnect, onSelectFolder, onOpenSettings`.

- **Logo block** — accent square + "AutoMark" wordmark, plus a small **"About the maker"** button (✚ icon) that toggles a popover reading **"This app was created by a Christian."** — an intentional personal touch, dismissed by click-away.
- **FILES section** — heading with a `change` link (only when `connected`) to re-pick the folder. Then either: a "Connect your files" CTA (not connected), an empty-state line (no subfolders), or the **subfolder list** — each a button with an active-state dot (`--accent-400` when selected) that calls `onSelectFolder`.
- **Bottom cluster:**
  - `<AllowanceBar/>` (Category 1 surface — % left, low-balance warning, links `/plans`).
  - **User row** — initials avatar (derived from `profileName`), name + subject, chevron. Opens a popover menu with **Settings** (→ `onOpenSettings`) and **Sign out**.
  - **Sign out** — `createClient().auth.signOut()` then `router.push("/login")` + `refresh()`. (Auth specifics → Category 7.)

## 5. Components catalogue

### 5.1 `SettingsPanel.tsx` — the settings slide-over (also the settings *model*)

This file is **double-duty**: it renders the panel **and** defines the canonical `Settings` type, defaults, and persistence used app-wide.

**Exported model:**
- `interface Settings { defaultStrictness, accent, profile{name,subject}, markTypes[], markingQuality, subjects[] }`
- `DEFAULT_SETTINGS` — strictness 7, accent `indigo`, profile `{name:"Michael Bernard", subject:"English"}`, `DEFAULT_MARK_TYPES`, quality `standard`, subjects `["English","Mathematics"]`.
- `DEFAULT_MARK_TYPES` — six exam-standard marks: **M** Full (green tick), **½** Half (green half), **A** Accuracy (blue tick), **✗** Incorrect (red cross), **E** Explanation (amber circle), **FT** Follow-through (cyan underline).
- `loadSettings()` / `saveSettings()` — read/write `localStorage["automark.settings"]`, with defensive merging against `DEFAULT_SETTINGS` (so older saved blobs don't break when the shape grows).

**Panel UI** (right slide-over, `max-w-md`, `slidein` keyframe, backdrop blur):
- **Profile** — display name + subject inputs.
- **Default marking strictness** — 1–10 range with a `n/10` pill.
- **Marking engine** — two cards, **Standard** vs **High accuracy**. *Model names are intentionally hidden* (the Sonnet/Opus mapping lives server-side, Cat 5); copy only says "Higher accuracy uses more of your allowance."
- **Mark types** — full CRUD: live `MarkShapeIcon` preview, native colour picker, abbrev (`maxLength 3`), label, remove, a 6-shape picker, and "Add mark type". This is what the PDF stamper (Cat 2) draws and what the marking prompt (Cat 5) lists.
- **Accent color** — five swatches (the `ACCENTS` array).
- **Account** — sign out.
- **Footer** — Cancel / Save. Save persists to localStorage, applies `data-accent`, calls `onSave`, closes.

> Note: `handleSave` preserves `initial.subjects` untouched — **subjects are managed elsewhere** (the combobox on the main page, §5.5), not in this panel.

### 5.2 `MarkShapeIcon.tsx` — on-screen shape renderer
Exports `MarkShape` type and `MARK_SHAPES` list (tick/half/cross/circle/underline/dot) and renders each as an inline SVG. Used for previews and the shape picker in SettingsPanel.

> ⚠️ **Two separate shape implementations exist.** This file draws shapes *on screen*; `drawShape()` in `lib/markPaper.ts` (Cat 2) draws them *onto the PDF* with different primitives. They must be kept **visually consistent by hand** — changing one doesn't change the other.

### 5.3 `AllowanceBar.tsx`
Category 1 surface (see that doc §8.1). Visually: plan label + "% left", progress bar, ≤15% red low-balance warning, "Buy a plan to start marking →" when cap is 0. Listens for the `allowance-refresh` window event.

### 5.4 `StrictnessSlider.tsx`
Standalone card with a 1–10 accent range and Lenient↔Strict end labels. Mirrors the SettingsPanel strictness control but is the **per-batch** value on the main page (seeded from `defaultStrictness`, overridable per run).

### 5.5 `SubjectCombobox.tsx`
Custom searchable combobox (no library). Filters `options` by query, supports **add-new** ("Add '<x>'"), click-away to close, and Enter to either add the typed value or pick the first match. New subjects flow up via `onAddOption` → `handleAddSubject` in `page.tsx` → persisted into `settings.subjects`.

### 5.6 `UploadZone.tsx` — ⚠️ DEAD CODE
A drag-and-drop / click file picker accepting `.pdf,.docx,.txt`. **Imported nowhere** (grep-confirmed) — a leftover from a pre-folder-based design. It also references `.docx`, which the app doesn't support. *Candidate for deletion, or the basis for the Phase-1 cross-browser picker* (its drag-drop pattern is exactly what `webkitdirectory` replacement could reuse).

## 6. PWA shell (install + update)

AutoMark is a **Progressive Web App** — installable, with a service worker and an update flow.

- **Manifest & metadata** — [`app/layout.tsx`](../../app/layout.tsx): `manifest: "/manifest.json"`, `appleWebApp` (capable, black-translucent, title "AutoMark"), `viewport.themeColor: "#4F46E5"`.
- **`InstallButton.tsx`** (header) — captures Chrome's `beforeinstallprompt`, suppresses the mini-infobar, and shows an "Install app" button **only when installable**. Hides if already installed (`appinstalled` event or `display-mode: standalone`).
- **`UpdatePrompt.tsx`** (rendered globally in `layout.tsx`) — registers `/sw.js`, detects a waiting service worker, shows a centered **"A new version is available. → Reload"** pill, posts `SKIP_WAITING`, and reloads once on `controllerchange`. This is the app's only persistent toast-style notification.

## 8. Notifications & Communication (folded-in)

### 8.1 What exists today
There is **no email or push notification infrastructure**. All current "communication" is in-app or manual:

| Channel | Mechanism | Where |
|---------|-----------|-------|
| In-app error/success | Red/green banners | `app/page.tsx` |
| **Block-reason notice** (why marking was refused) | Red banner via `blockMessage()` + persistent `PlanNotice` | `app/page.tsx`, `PlanNotice.tsx` (see §8.4) |
| Progress | Text inside the Mark button | `app/page.tsx` |
| App-update toast | Service-worker `UpdatePrompt` pill | global (`layout.tsx`) |
| Outbound to founder | WhatsApp deep-links (payment proof) | `app/plans/page.tsx` |
| **Ops alert to founder** | **`notifyOps()` → webhook push (e.g. ntfy.sh) + log** | `lib/notify.ts` (see §8.3) |
| Sign-up confirmation | Supabase confirmation email (if enabled) | Supabase (Cat 6/7) |

### 8.2 What's planned (not built)
Per `../expansion-plan.md` (Phases 2–4) and Category 1:
- **Batch-complete email** — "Your N papers are marked" (so lecturers needn't keep the tab open for the ~20-min poll).
- **Trial-expiry warning** — a heads-up a day before the 7-day trial / R50 cap cuts off.
- **Renewal / payment confirmation** — once automated billing (PayFast) lands.
- **Channel of record:** Resend (generous free tier, good SA delivery). None implemented yet.

### 8.3 Ops alerts — `notifyOps()` (BUILT 2026-06-15)

The first **real outbound notification channel** in the app. It alerts the founder (not the end-user) when something operational fails — currently (a) a Supabase usage-**write** failure (Category 1, Problem 8) and (b) a usage-**read/verify** failure that blocked a user from marking (Category 1, P1-2 — `checkAllowance()`).

- **Code:** `lib/notify.ts` — `notifyOps(message)`. Always `console.error`s (so it's in Cloudflare observability), and if the `OPS_ALERT_WEBHOOK_URL` secret is set, `POST`s the message there.
- **Channel:** a plain-text webhook. Designed for a free **ntfy.sh** topic (subscribe in the ntfy app → phone push). Slack/Discord work too, but expect a JSON body (`{"text":…}` / `{"content":…}`) — tweak the `fetch` in `notify.ts` if using those.
- **Setup:** `npx wrangler secret put OPS_ALERT_WEBHOOK_URL` → paste the topic URL. **Until set, it's log-only** — no breakage, just no push.
- **Fired from:**
  1. `recordUsage()`'s final-failure path (`lib/usage.ts`), alongside parking the event in the D1 dead-letter buffer (Cat 1 §11b/§13b). *Message:* `Supabase usage write failed — parked R… …`.
  2. `checkAllowance()`'s verification-failure paths (`lib/usage.ts`, Cat 1 §6.3) — when an auth lookup or profile read errors and a user is therefore **blocked from marking**. *Message:* `Marking blocked: could not read allowance for <user> — …`. (The normal over-limit case does **not** page — that's expected, not a fault.)
- **Not yet wired:** end-user notifications (batch-complete, trial-expiry) in §8.2 are still unbuilt. `notifyOps` is reusable for any future ops alert.

### 8.4 User-facing block & plan notifications (BUILT)

When marking is refused, the user is **always told why** — never left with a silent no-op or a raw error code. These are the user-facing counterpart to the §8.3 ops alerts. The *decision* logic lives in Category 1 (`lib/allowance.ts` + `checkAllowance()`); the *presentation* lives here.

| Trigger | What the user sees | Surface / code |
|---------|--------------------|----------------|
| **Marking blocked by a system error** (auth/DB couldn't be verified — fail-closed, P1-2) | Red banner: *"We couldn't verify your plan just now (a temporary system error), so nothing was marked. Please try again in a moment — you weren't charged."* | `blockMessage("verification_failed")` → error banner in `app/page.tsx` |
| **Session expired** | Red banner: *"Your session has expired. Please sign in again to keep marking."* | `blockMessage("not_authenticated")` |
| **Allowance used up** | Red banner: *"You've used up your plan's allowance. Buy another plan to keep marking."* | `blockMessage("allowance_exhausted")` |
| **Plan expired / limit reached** (persistent, not tied to a mark attempt) | Top-of-page banner *"Your plan has expired / You've reached your plan limit"* + **Buy a new plan** button | `components/PlanNotice.tsx` (Cat 1 §8.3) |
| **Plan expiry date** (informational) | Settings → Plan: *"Renews/expires on &lt;date&gt;"* | `components/SettingsPanel.tsx` (Cat 1 §8.4) |

`blockMessage()` is the single mapping from a route error code to a human sentence — add new block reasons there so the user never sees a bare code.

## 9. Cross-cutting UX behaviours

- **Hydration safety** — `page.tsx` assumes `supported=true` during SSR, then re-checks `isSupported()` after mount via a `mounted` flag, to avoid a server/client mismatch on the File System Access API check.
- **Flash-free theming** — accent applied before paint (§2).
- **Persistence split** — settings & accent in `localStorage`; folder handles & memos in IndexedDB (Cat 2). Nothing user-specific is server-stored except the profile/usage rows (Cat 1).
- **Event bus (lightweight)** — the only cross-component signal is the `allowance-refresh` `window` event fired after marking, which tells `AllowanceBar` to re-fetch.
- **Strictness lives in two places** — the per-batch `StrictnessSlider` (seeded from `defaultStrictness`) and the SettingsPanel default. Changing the panel default reseeds the page slider on save.

## 10. Known gaps & issues (UI)

- **No onboarding / help system.** First-run is just the empty "Connect your files" state — no guided tour, no help button. This is **Phase 1** of the expansion plan (big help button on first launch → shrinks to a corner). Currently the single biggest UX gap.
- **`UploadZone.tsx` is dead code** (and references unsupported `.docx`). Delete or repurpose.
- **Two shape implementations** (`MarkShapeIcon` on screen vs `drawShape` on PDF) drift independently.
- **Chrome/Edge-only** is surfaced as an amber notice, but non-Chromium users have no path forward until the Phase-1 picker change (Cat 2).
- **No notification infrastructure** (§8.2) — lecturers must watch the tab during batch runs.
- **Wording is partially lecturer-friendly** ("Memo (answer key)", "Marking engine" instead of model names) but a full copy audit is still a Phase-1 item.
- **Accent/subjects coupling is implicit** — adding an accent needs paired edits in CSS + `ACCENTS`; subjects are edited in the combobox, not Settings. Easy to trip over.
